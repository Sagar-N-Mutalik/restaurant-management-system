Software Design Patterns

A behind-the-scenes look at the architecture powering our Restaurant Management System.

The Concept: Provides a simplified, unified interface to a complex subsystem.

How we use it: When a customer clicks "Submit Order", a lot happens: saving items to the database, calculating totals, verifying stock, and notifying the kitchen. Instead of the UI making 10 different function calls, it simply calls our OrderFacade.place_order(). The Facade hides the complex backend wiring from the frontend.

The Concept: Provides an interface for creating objects, but lets subclasses decide which class to instantiate.

How we use it: We have different types of menu items (Pizzas, Burgers, Sides). The ItemFactory is responsible for creating these specific item objects safely. If we introduce a new category (like "Desserts"), we just add it to the factory without rewriting the core ordering logic.

The Concept: Attaches additional responsibilities or behaviors to an object dynamically without altering its structure.

How we use it: Customizations! When a user clicks "+ Extra Cheese" or "+ Add Bacon", we don't have a separate database item for "Pizza with Cheese and Bacon". Instead, we use a Decorator to wrap the base Pizza object, dynamically adding ₹50 or ₹100 to the price and updating the description on the fly.

The Concept: A subscription mechanism to notify multiple objects about any events that happen to the object they're observing.

How we use it: The Kitchen Dashboard is an Observer. When the Customer UI places an order (the Subject state changes), it automatically notifies the Kitchen Observer to update the Chef's screen, ensuring real-time synchronization without the frontend having to manually ping the kitchen.

The Concept: Defines a family of algorithms, encapsulates each one, and makes them interchangeable at runtime.

How we use it: Pricing and billing rules. If a customer is getting a "Happy Hour Discount" versus standard pricing, or if they are paying via different gateways, the Strategy pattern allows us to swap out the calculation logic instantly without writing massive `if/else` statements in the checkout code.

The Concept: Ensures a class has only one instance and provides a global point of access to it.

How we use it: Used for our central Kitchen State Manager or Restaurant Configuration. We only want *one* central brain keeping track of active table states; otherwise, duplicated managers would cause overlapping orders and corrupted table statuses.
